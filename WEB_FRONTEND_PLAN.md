# SameTime 前端 Web 版本详细计划表

## 1. 项目现状速读（用于规划依据）

- 现有前端是 **Vue 3 + Pinia + Vue Router + Vite**，UI 与业务状态已经较完整。Web 版可复用路由、页面、房间/用户/同步状态管理。  
- 当前播放能力强依赖 **Tauri + MPV**：`player store` 通过 `invoke` 调用 Rust 命令控制 MPV；`VideoPlayer` 使用 Tauri 文件选择与本地文件 URL 转换。这是 Web 化的最大阻塞点。  
- 服务端已具备 Web 可复用能力：认证、房间、文件上传下载、WebSocket 信令均为标准 HTTP/WS API。  


### 1.1 新前置条件（本次修订）

- 本项目 Web 版采用**本地同源文件模式**：所有参会用户在本地已持有同一视频源文件。  
- 服务端职责是“状态同步与房间协作”，而非“媒体分发”。  
- 因此前端播放方案优先优化：本地文件选择、hash 一致性提示、同步精度与播放兼容。  

---

## 2. Web 版本目标与范围

### 2.1 目标（MVP）

1. 支持浏览器直接进入房间、聊天、成员状态同步、基础语音通话。  
2. 支持浏览器原生 `HTMLVideoElement` 播放（以本地文件为唯一播放源）。  
3. 保留“同步观影核心体验”：播放/暂停/跳转同步、漂移校准、文件一致性提示（可降级实现）。  

### 2.2 非目标（首期不做）

- 不做浏览器内磁力下载（浏览器安全模型不友好）。  
- 不做“服务器分发视频文件”能力（本项目约束为所有成员本地已有同一视频源）。  
- 不追求与桌面端完全一致的播放器能力（如 MPV 级格式兼容）。  
- 不做复杂 DRM/转码服务（后续演进）。

## 3. 技术方案（推荐）

### 3.1 前端架构改造

引入“播放器适配层”，从 `player store` 中抽离平台差异：

- `PlayerAdapter` 接口：`load/play/pause/seek/setVolume/getState/dispose`。
- `TauriMpvAdapter`：复用现有调用。
- `WebVideoAdapter`：基于 `<video>` 与媒体事件。
- 运行时选择策略：
  - 桌面端（Tauri 环境）→ `TauriMpvAdapter`
  - 浏览器 → `WebVideoAdapter`

> 这样可做到“一套业务同步逻辑 + 两套底层播放实现”。

### 3.1.1 前端播放能力技术方案（重点完善）

为了让 Web 版既可快速上线，又能平滑演进，建议采用 **三层播放架构**：

1. **播放内核层（Playback Engine）**
   - `NativeVideoEngine`：基于浏览器原生 `<video>`，优先使用，成本最低。
   - `HlsEngine`（可选）：当后端提供 `.m3u8` 时通过 `hls.js` 扩展兼容（特别是非 Safari 浏览器）。
   - `DashEngine`（二期可选）：若后续需要多码率自适应，可评估 `dash.js`/Shaka。

2. **适配层（PlayerAdapter）**
   - 屏蔽 Tauri/Browser 差异，向上暴露统一能力。
   - 统一事件模型：`onTimeUpdate/onPlayState/onError/onEnded/onBuffering`。

3. **业务同步层（Sync Controller）**
   - 只消费“标准播放器事件”，不感知底层是 MPV 还是 `<video>`。
   - 管理远端消息、本地交互优先级、漂移校准与冲突消解。

#### A) MVP 阶段播放源优先级

- **P0（首发必做）**：本地文件（`File` + `URL.createObjectURL`）
- **P1（可选）**：仅用于兼容联调的远端直链调试源（默认关闭，不作为产品主流程）
- **P2（长期可选）**：HLS/DASH（仅在产品方向变化为“服务端分发流媒体”时启用）

> 结论：以“本地文件 + 原生 `<video>`”作为唯一主链路，优先保证同步准确性与低运维复杂度。

#### B) 编解码与容器策略（必须写入产品约束）

- Web 首发“官方支持”建议限定：
  - 容器：`mp4`
  - 视频编码：`H.264 (AVC)`
  - 音频编码：`AAC`
- 对 `mkv/flv/wmv` 等高风险格式：
  - UI 给出“可能无法播放”提示；
  - 提供离线转码建议（FFmpeg 命令示例可补在 FAQ）。

#### C) Web 播放与同步关键参数

- 时间采样频率：250ms（UI 显示）/ 500ms（网络广播）
- 漂移阈值：
  - `< 0.8s`：忽略
  - `0.8s ~ 2.0s`：轻微校准（`playbackRate` 0.95~1.05）
  - `> 2.0s`：强制 `seek`
- 防抖与风暴保护：
  - `seek` 后 800ms 内忽略远端二次 `seek`
  - 1 秒内最多发送 2 次同步广播

#### D) 播放失败与降级路径

当 `<video>` 触发 `error` 时，按顺序执行：

1. 校验本地文件扩展名与 MIME 是否匹配
2. 通过 `video.canPlayType` 检查浏览器能力并给出明确错误文案
3. 若用户启用了远端调试源且为 HLS，原生不支持时尝试 `hls.js` 回退
4. 给出“转码后再试”与“重新选择本地文件”两个兜底动作

#### E) 性能与内存治理（Web 必做）

- 本地文件切换时立即 `URL.revokeObjectURL`，避免内存泄漏。
- 大文件 hash 放入 `Web Worker`，主线程只接收进度。
- 房间长会话场景（30~60 分钟）监控：
  - `JS Heap` 增长曲线
  - 卡顿率（长任务 > 50ms）
  - 重连次数与恢复时延

#### F) 安全与策略

- CSP 白名单仅开放必要媒体域名。
- 生产环境强制 HTTPS/WSS（否则部分浏览器会限制媒体与麦克风能力）。
- 本地文件选择前后做大小/后缀提示校验；若保留上传接口仅作为运维通道，后端继续做魔数校验（双保险）。

#### G) 落地建议（可直接分配任务）

1. 新增 `src/player/adapters/webVideoAdapter.ts`（封装 `<video>` 事件与控制）。
2. 新增 `src/player/core/syncController.ts`（统一同步策略参数）。
3. `VideoPlayer.vue` 改造为“渲染层”，不直接依赖 Tauri API。
4. `player store` 仅编排状态，不直接调用 `invoke`。
5. 增加 `playback-capability` 诊断面板（格式支持、自动播放策略、音频设备状态、文件一致性校验状态）。

### 3.2 文件能力降级策略

- 本地文件：Web 通过 `<input type="file">` + `URL.createObjectURL`。
- 文件 hash：
  - 中小文件：浏览器内 `crypto.subtle.digest` 分片 hash（与桌面采样策略对齐）。
  - 超大文件：采样 hash（头/中/尾）保持一致性方案。
- 上传能力：在 Web 版中默认关闭（仅桌面端/运维工具保留），避免与“本地同源文件”前置条件冲突。

### 3.3 语音通话

- 当前 `useVoice.ts` 基于 WebRTC，可在 Web 直接复用。
- 增加浏览器权限与设备选择 UI（麦克风授权失败可恢复）。

### 3.4 鉴权与部署

- 首期保持 JWT + localStorage（与现有一致）。
- 增加 `VITE_API_BASE_URL`、`VITE_WS_BASE_URL`，避免写死 localhost。
- Nginx 反向代理 `/api` 与 `/ws`，解决跨域和 WSS。

### 5.1 子任务拆解（面向 Agent 可直接执行）

> 说明：以下将 5.x 中每个任务拆成“输入 → 实施步骤 → 完成定义（DoD）”，Agent 可按编号逐项提 PR。

#### A. 基础设施

**A1. 新增 `src/platform/`：平台探测、环境变量读取、运行时能力标记**
- 输入：现有前端入口、`localStorage` 读取逻辑、Vite 环境变量。
- 子任务：
  1) 新建 `src/platform/runtime.ts`，导出 `isTauri/isWeb` 能力探测。
  2) 新建 `src/platform/config.ts`，统一读取 `VITE_API_BASE_URL`、`VITE_WS_BASE_URL`，提供默认值。
  3) 新建 `src/platform/capabilities.ts`，聚合 `canUseLocalFile/canUseWebRTC/canUseFullscreen`。
  4) 在 `main.ts` 或 `App.vue` 初始化时打印一次 capability 摘要日志（仅开发环境）。
- DoD：业务代码不再直接散落读取 `import.meta.env`；平台判断只从 `platform/*` 获取。

**A2. 抽象播放器能力接口与错误码**
- 输入：`player store` 当前接口和状态字段。
- 子任务：
  1) 新建 `src/player/types.ts`：定义 `PlayerAdapter`、`PlayerState`、`PlayerErrorCode`。
  2) 新建 `src/player/errors.ts`：统一错误映射（如 `UNSUPPORTED_CODEC`、`FILE_NOT_FOUND`、`PLAY_ABORTED`）。
  3) 为 `PlayerAdapter` 定义事件订阅方法（`onTimeUpdate/onStateChange/onError`）。
  4) 写最小类型测试/编译检查（保证 Adapter 实现必须满足接口）。
- DoD：`player store` 能只依赖接口类型，不依赖具体实现细节。

**A3. 增加统一日志/埋点封装（播放器事件、ws 重连、同步偏差）**
- 输入：现有 `console.log`、WebSocket 重连逻辑。
- 子任务：
  1) 新建 `src/utils/telemetry.ts`（`track(event, payload)`）。
  2) 将播放器关键事件接入（load/play/pause/seek/error）。
  3) 将 WS 重连事件接入（attempt/success/fail）。
  4) 将同步偏差统计接入（drift value、校准动作类型）。
- DoD：核心路径的关键行为都可被结构化记录，且生产可关闭 debug 日志。

#### B. 播放器改造

**B1. `player store` 从直接 `invoke` 改为调用 adapter**
- 输入：当前 `src/stores/player.ts`。
- 子任务：
  1) 抽离原有 MPV 调用到 `TauriMpvAdapter`。
  2) 新建 `WebVideoAdapter` 并对齐 `load/play/pause/seek/setVolume`。
  3) 在 store 中注入 adapter 实例（工厂模式按平台选择）。
  4) 保留原 `syncTo` 逻辑，但改为调用 adapter 状态而非 `invoke`。
- DoD：`player.ts` 不再直接 import `@tauri-apps/api/core`。

**B2. `VideoPlayer.vue` 去除硬编码 Tauri API，按平台渲染**
- 输入：当前 `VideoPlayer.vue`。
- 子任务：
  1) 将文件选择封装为 composable：`useVideoSourcePicker`。
  2) Web 端渲染 `<video ref="videoEl">`，桌面端保留 MPV 容器。
  3) 将“选择本地文件/错误提示/兼容性提示”拆成子组件。
  4) 保持现有 hash 提示 UI 行为不退化。
- DoD：组件内不直接出现 Tauri 特定 import（仅桌面专属模块内部允许）。

**B3. 增加资源释放（离房、切源时 `revokeObjectURL`）**
- 输入：本地文件 object URL 生命周期。
- 子任务：
  1) 维护 `currentObjectUrl` 状态。
  2) 切换文件前先 `URL.revokeObjectURL(old)`。
  3) 离房、组件卸载时执行统一 `dispose()`。
  4) 增加防重复释放与空值保护。
- DoD：连续切换 10 次本地大文件后内存无明显线性增长。

#### C. 同步机制

**C1. 统一“本地操作”与“远端同步”优先级**
- 输入：当前 `sync_broadcast` 处理。
- 子任务：
  1) 设计优先级规则：本地用户操作窗口内优先本地。
  2) 为每次操作打 `origin`（local/remote）和 `opId`。
  3) 忽略重复消息与过期消息。
  4) 冲突时记录 telemetry 便于回放分析。
- DoD：不会出现“本地刚 seek 又被远端立即拉回”的频繁抖动。

**C2. 增加防抖窗口（避免 seek 风暴）**
- 输入：同步广播频率与 seek 触发逻辑。
- 子任务：
  1) 对本地拖动进度条做节流（如 300ms）。
  2) 对远端 seek 应用冷却窗口（如 800ms）。
  3) 引入“最后一次有效 seek 时间戳”。
  4) 对超过阈值的高频消息进行丢弃并记录。
- DoD：双端快速拖动不会造成持续 seek 循环。

**C3. 把同步参数配置化（阈值、广播频率）**
- 输入：文档内建议参数。
- 子任务：
  1) 新建 `src/player/syncConfig.ts`。
  2) 将漂移阈值、广播周期、冷却时间抽常量。
  3) 支持开发环境覆盖配置（便于 A/B 调参）。
  4) 在日志输出当前生效参数。
- DoD：参数调整不需要改业务逻辑代码。

#### D. 媒体与文件

  1) 新建 `src/utils/hashWorker.ts` 与 worker 脚本。
  2) 实现头/中/尾采样读取与 SHA-256 计算。
  3) 增加进度回调（0~100%）并可取消任务。
  4) 与房间 `video_hash` 消息打通比对结果。
- DoD：同一文件在桌面端与 Web 端 hash 结果一致。

**D2. 本地文件选择失败重试和大小/格式前置校验**

## 6. 风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 浏览器编码格式支持不一致 | 某些本地视频无法播放 | 明确支持格式白名单（mp4/h264/aac），给出转码建议 |
| WebRTC 在企业网络受限 | 语音失败率上升 | 增加 TURN 配置入口与失败回退提示 |
| 大文件 hash 耗时长 | UI 卡顿、体验差 | Web Worker + 分片计算 + 进度提示 |
| 同步抖动 | 观影体验下降 | 引入校准节流与漂移窗口策略 |
| WebSocket 断线 | 房间状态丢失 | 断线重连、状态重拉、幂等消息处理 |
